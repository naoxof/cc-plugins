# tmplt

## 概要

tmpltは、`<CLAUDE>`プレースホルダーを含むMarkdownファイルを、インタラクティブなヒアリングを通じて埋めていくClaude Codeプラグインです。

## 機能

- ファイル内の`<CLAUDE>`プレースホルダーを埋めるためのヒアリングと推論を実施
  - 技術情報は自動収集、ユーザーの判断が必要な部分だけをヒアリング
- 周辺コンテキストを読み取って適切なコンテンツを起草
- `creator`スキルを使って`<CLAUDE>`プレースホルダーを含むMarkdownファイルを生成することも可能

## 使い方

### 基本: テンプレートを埋める

例えば、以下のようなMarkdownファイルを用意します（`proposal.md`）：

```markdown
# プロジェクト提案書

## 概要
<CLAUDE: プロジェクトの目的と期待される成果を簡潔に記述>

## 背景
<CLAUDE: 現在の課題とこのプロジェクトが解決する問題>

## 提案内容
<CLAUDE: 具体的なアプローチを箇条書きで説明>

## 次のステップ
<CLAUDE: 承認後のアクションアイテム>
```

そして`/tmplt:filler`スキルを実行します：

```bash
/tmplt:filler proposal.md
```

Claudeは各`<CLAUDE>`プレースホルダーについて：
1. 周辺のコンテキストを分析して何を書くべきか理解
2. 技術的な情報は自動で収集（コード検索、Web検索など）
3. ユーザーの判断が必要な部分だけをヒアリング
4. 回答を基にコンテンツを起草
5. プレースホルダーを実際の内容で置き換え

すべてのプレースホルダーが埋められた完成版ドキュメントが出来上がります。

### オプション: creatorでテンプレートを作成

テンプレートを自分で作るのが面倒な場合、`/tmplt:creator`スキルに作ってもらうこともできます：

```bash
/tmplt:creator proposal.md
```

Claudeがインタラクティブに要件をヒアリングし：
- ドキュメントの種類・目的
- 対象読者
- 必要なセクション・構成
- スタイルとトーン

適切な`<CLAUDE>`プレースホルダーを配置したテンプレートを自動生成します。その後、上記の`/tmplt:filler`で埋めるだけです。

## ユースケース

### 複雑なドキュメントを作成する場合

シナリオ: プロジェクト提案書を書きたいが、どんなセクションが必要か分からない

通常、このようなドキュメントを作る際によくある悩み：
- どんなセクションを入れるべきか構成が分からない
- 各セクションで何を書けばいいか判断に迷う
- 検討して書き込む箇所が多くて大変
- 回答を出すためにどのような観点で考えれば良いのかわからない

tmpltでの解決方法：

```bash
/tmplt:creator project-proposal.md
```

Claudeがヒアリング：
- 提案の種類は？（新規プロジェクト、予算申請、組織変更など）
- 対象読者は？（経営層、部門長、チームメンバーなど）
- 必要な要素は？（予算、スケジュール、リスク分析など）

あなたの要件に最適化された構成のテンプレートが生成されます。一旦テンプレートを確認して不要な箇所を削ったり、セクションを追加することも可能です。
その後fillerを呼び出します：

```bash
/tmplt:filler project-proposal.md
```

fillerの価値:

ここからが重要です。fillerは各プレースホルダーについて、あなたが考えるべき観点を具体的な選択肢として提示します。

例えば「プロジェクトの目標」を書くべき箇所では：
- 質問: このプロジェクトの主な目標は？
  - A) コスト削減（説明: 運用コストを20-30%削減することを重視）
  - B) 売上増加（説明: 新規顧客獲得や既存顧客単価向上を重視）
  - C) 業務効率化（説明: 作業時間の短縮や自動化を重視）
  - D) リスク低減（説明: セキュリティやコンプライアンス対応を重視）

このように、「何を考えれば良いか」という観点を選択肢で示してくれるため：
- 白紙から考える必要がなく、選択肢を見て「これだ」と判断できる
- 選択肢にない場合は自由記述で補足できる
- 各選択肢の説明を読んで、自分の考えを整理できる
- 検討すべき観点が網羅的に提示されるので漏れがない

さらに、fillerは：
- 技術的な裏付けは自動で調査
- 業界のベストプラクティスをWeb検索で収集
- あなたの回答を基に、プロフェッショナルな文章を起草
- 書式は自動で整える

メリット:
- ドキュメント設計から考える手間が省ける
- 目的に合った構成が自動で提案される
- 「何を考えればいいか」が選択肢で示されるので迷わない
- 検討すべき観点が網羅的に提示されるので漏れがない
- 選択肢を選ぶだけで、プロフェッショナルな文章が生成される
- 大量の記述作業から解放される

## プレースホルダーの種類

### 基本プレースホルダー
```markdown
<CLAUDE>
```
周辺のコンテキストから内容を推測させる場合に使用。

### ヒント付きプレースホルダー
```markdown
<CLAUDE: ここにユーザーの目標や成功基準を記述>
```
何を書くべきか明確にガイダンスを提供します。

## 設計の特徴

### インタラクティブなヒアリング
すべてのユーザーとのやり取りは`AskUserQuestion`ツールを使用し、選択肢ベースの対話で進みます。自由記述も可能です。

### 技術情報の自動収集
fillerスキルは以下を自動的に収集します：
- プロジェクト内のコードベース検索
- 既存パターンの分析
- Web検索による最新情報の取得

ユーザーにはビジネス判断や好みなど、「ユーザーだけが知っていること」のみを質問します。

### トレンド対応
技術スタック、ライブラリ、製品などトレンドが変わる分野では、必ずWeb検索で最新情報を収集してから選択肢を作成します。

## ベストプラクティス

### テンプレート設計時
1. 適切な粒度でプレースホルダーを配置 - 段落や論理的なセクション単位で
2. コンテキストを提供 - プレースホルダーの前後に説明文を配置
3. ヒントを効果的に使う - 「3つの主要な機能と各々のメリットを記述」など具体的に
4. 階層構造を明確に - 見出しレベルを適切に使用

### テンプレート利用時
1. 段階的に埋める - 1つずつプレースホルダーを確認しながら進める
2. 下書きをレビュー - Claudeが起草した内容を確認し、必要に応じて修正を依頼
3. テンプレートを改善 - 使ってみて分かった問題点を次回のテンプレートに反映

## 対応ドキュメント種類

- 技術ドキュメント: API仕様、設計書、実装ガイド
- ビジネスドキュメント: 提案書、報告書、計画書
- クリエイティブコンテンツ: ブログ記事、マーケティング資料
- その他、構造化されたドキュメント全般

## よくある質問

### Q: テンプレートは自分で作る必要がありますか？
A: 自分で作っても良いですし、creatorスキルに作ってもらってもOKです。`<CLAUDE>`を含むMarkdownファイルなら何でも使えます。

### Q: 既存のMarkdownファイルにも使えますか？
A: はい！既存のファイルに `<CLAUDE>` を挿入すれば、その部分だけを埋めることができます。

### Q: 途中でスキップしたいプレースホルダーがあります
A: fillerスキルにスキップしたい旨を伝えれば、先に進んで後で対応できます。

### Q: 起草された内容が気に入りません
A: その場で修正を依頼できます。Claudeは即座に修正します。

### Q: プレースホルダーにヒントを付けるべきですか？
A: 推奨します。`<CLAUDE: 具体的な指示>`の形式でヒントを付けると、より適切なコンテンツが生成されます。ヒントなしの`<CLAUDE>`でも周辺のコンテキストから推測しますが、明示的な方が確実です。

### Q: creatorで作ったテンプレートは再利用できますか？
A: はい！一度作成したテンプレートは何度でも使えます。コピーしてfillerで埋めるだけです。
